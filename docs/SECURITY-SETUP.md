# Налаштування безпеки після аудиту

Цей документ описує, що змінилось у безпеці проєкту та які **ручні кроки**
потрібно виконати, щоб активувати виправлення. Поки кроки не виконані,
сайт працює як раніше (нічого не ламається), але критична вразливість
залишається відкритою.

---

## Що було не так

1. **Адмінка не мала реальної автентифікації.** Пароль `VITE_ADMIN_PASS`
   вшивався у публічний JS‑бандл, а всі привілейовані операції (схвалення,
   видалення, редагування подій/бірж/типів) виконувались через **публічний
   anon‑ключ**. Захист тримався лише на RLS, який мусив дозволяти anon‑записи,
   щоб панель працювала, — тобто **будь‑хто з anon‑ключем (а він публічний) міг
   напряму змінювати/видаляти будь‑які дані**.
2. Cron‑ендпоінти були або без автентифікації, або з «fail‑open» перевіркою
   (`delete-old-events` при незаданому секреті відкривав масове видалення усім).
3. Не було HTTP security‑заголовків; кілька дрібніших проблем (див. нижче).

---

## Що зроблено в коді

- **Supabase Auth для адмінки** (`src/pages/Admin.jsx`): вхід тепер через
  email+пароль, а доступ підтверджується RPC `is_admin()`.
- **Жорсткий RLS** (`supabase/migrations/20260624120000_rls_lockdown.sql`):
  - anon: лише читання публічних даних + подача заявок/правок + реакції;
  - admin (рядок у `public.admins`): повний доступ;
  - cron/Edge Functions працюють через service‑role (RLS їх не стосується).
- **Cron‑автентифікація** через спільний `scripts/lib/cronAuth.js` (вимагає
  `CRON_SECRET`; Vercel Cron надсилає його автоматично).
- **HTTP‑заголовки** (CSP, X‑Frame‑Options, HSTS тощо) у `vercel.json`.
- Кнопка «Оновити статистику» більше не пише в БД з браузера — вона викликає
  серверну Edge Function `price-reaction-cron`.
- Прибрано `window.supabase`; посилено валідацію `mexc-ticker`; екранування `"`
  у Telegram‑постах.

## Ручні кроки (виконати в такому порядку)

> ⚠️ Публічний сайт працює протягом усього процесу. Коротка недоступність
> можлива лише для самої адмін‑панелі — у вікні між деплоєм і додаванням вашого
> акаунта в `public.admins`.

### 1. Задати `CRON_SECRET` у Vercel (обовʼязково перед деплоєм цієї гілки)

Project → Settings → Environment Variables → додати:

```
CRON_SECRET = <довгий випадковий рядок, напр. `openssl rand -hex 32`>
```

Без нього cron‑ендпоінти повертатимуть 401 (це і є захист). Vercel Cron
надсилає цей секрет автоматично.

### 2. Створити адмін‑акаунт у Supabase

Dashboard → Authentication → Users → **Add user** → вказати ваш email і пароль
(підтвердити email).

Там само вимкнути самостійну реєстрацію, щоб ніхто чужий не міг створити акаунт:
Authentication → Sign In / Providers → Email → вимкнути **Allow new users to
sign up** (Confirm signups).

### 3. Задеплоїти цю гілку у production

Змерджити/задеплоїти `claude/adoring-bardeen-mhrkmy`. Нова адмінка (Supabase
Auth) ще не пустить вас, поки не застосовано міграцію (крок 4) — це нормально.

### 4. Застосувати міграцію RLS

Будь‑який зі способів:

- **Supabase SQL Editor:** вставити вміст
  `supabase/migrations/20260624120000_rls_lockdown.sql` і виконати; **або**
- **CLI:** `supabase db push`; **або**
- попросіть мене застосувати її через MCP (`apply_migration`) — я зроблю це,
  щойно ви підтвердите, що кроки 2–3 виконані (проєкт треба перевести з
  INACTIVE у активний).

### 5. Додати ваш акаунт у `public.admins`

У SQL Editor:

```sql
insert into public.admins (user_id)
select id from auth.users where email = 'ВАШ_EMAIL'
on conflict (user_id) do nothing;
```

Після цього вхід в адмінку (email+пароль) працює повноцінно.

### 6. Прибрати `VITE_ADMIN_PASS`

Більше не використовується — видаліть змінну з Vercel.

## Перевірка після активації

- Відкрити сайт **в анонімному вікні** і в консолі браузера спробувати:

  ```js
  // Має повернути дані (читання дозволене):
  await window.__sb?.from?.('events_approved').select('*').limit(1)
  // А запис має відхилятися RLS (помилка / 0 рядків):
  await window.__sb?.from?.('events_approved').delete().neq('id','0')
  ```

  (`window.supabase` навмисно прибрано; перевіряйте через свій клієнт.)
- Зайти в `/admin` без входу → доступу немає; після входу email+пароль →
  панель працює.
- Переконатися, що cron‑джоби у Vercel виконуються успішно (200, не 401).

## Залишковий ризик (до уваги)

Реакції (лайки/дизлайки) використовують згенерований у браузері `anon_id`, який
не є перевіреною особою. RLS не може привʼязати ці рядки до реального
користувача, тому маніпуляція лічильниками реакцій технічно можлива (дані
несенситивні). За потреби — обмежити серверною ідентичністю/рейт‑лімітом.
